# v4.0 Source Scope for Second Exhibition

> Scope of this document: define, *before any download*, which source types are eligible for the second real exhibition, which institutions are plausible candidates for research, and what metadata must accompany every future asset. Companion to `docs/SECOND_EXHIBITION_PLAN_v4.0.md`.

---

## Purpose

A second exhibition's most expensive failure mode is **scope and rights drift**: assets are gathered from search results or memory, rights are unclear, credit lines are vague, and the release has to be unwound. The v3.4 hardening cycle exists precisely to prevent this.

This document therefore fixes the **input surface** first:

1. Which **source types** are accepted.
2. Which **institutions** are candidate sources (research only — no download in v4.0).
3. Which **metadata fields** every future asset must carry.
4. What the **source acceptance rule** is, before an asset may enter a formal exhibition.

The four rules together form the v4.0 *source gate*. Anything that does not pass the source gate is excluded from build rounds v4.3 and v4.4.

---

## Preferred source types

The following source types are **eligible** for the second exhibition, in priority order. The list is intentionally narrow — narrowness is the point.

| Priority | Source type | Why it is preferred |
|---|---|---|
| 1 | **Open access museum collections** | Clear rights statements, stable URLs, persistent identifiers. |
| 2 | **Public domain botanical illustrations** | Mostly pre-1928 works; rights cleanly public domain in most jurisdictions. |
| 3 | **Digitized books with clear rights statements** | Books whose digitization rights have been explicitly cleared (e.g., CC0, CC-BY, public domain mark). |
| 4 | **Institution-provided IIIF images if rights allow** | IIIF manifests expose rights metadata as a first-class field — verifiable per asset. |
| 5 | **Project-generated diagrams** | Diagrams, comparisons, and explanatory figures generated by the project itself; rights owned by the project. |

The following are **explicitly out of scope** for v4.0 and v4.3:

- Search-result images imported without a traceable origin page.
- Images imported "from memory" of an earlier browsing session.
- Screenshots of third-party platforms where the rights status of the underlying image is not separately verifiable.
- AI-generated images whose training data and rights are not auditable.
- Software UI screenshots whose license is unclear.

The exclusion list is as important as the inclusion list. It is the negative space that keeps the source gate small enough to defend.

## Candidate institutions

The following institutions are **research candidates** for v4.1 (source-candidate research). They are listed here to fix the *candidate surface* — v4.0 does not contact them, does not crawl them, and does not download from them.

| Institution | Why it is on the candidate list | What v4.1 will check |
|---|---|---|
| **Biodiversity Heritage Library (BHL)** | Large corpus of public-domain botanical and natural-history literature; clear rights posture; API and IIIF available. | Per-page rights statement; download terms; IIIF manifest availability. |
| **Wellcome Collection** | Open-access medical and historical imagery; CC-BY on many items; clear credit-line conventions. | Per-item license; required attribution text; IIIF availability. |
| **Smithsonian Open Access** | CC0 on a large portion of holdings; stable identifiers; bulk metadata. | Per-asset license; bulk vs. per-item API access; identifier persistence. |
| **The Met Open Access** | CC0 on a substantial portion of collection; public-domain + public-domain-mark split must be checked per item. | Per-object rights line; image URL vs. object URL; usage restrictions. |
| **Rijksmuseum** | Clear open-data policy; IIIF available; per-asset rights metadata. | Per-asset rights line; IIIF; required credit text. |
| **Library of Congress** | Public-domain corpus with stable identifiers; digital collections pages are rights-clear for most pre-1928 works. | Per-item rights line; persistent URL; IIIF availability. |

v4.0 does **not** commit to any of these as a final source. v4.1 will research them one by one and produce a candidate source table. v4.0 only commits to *considering* them.

## Required metadata per asset

Every future asset that enters a real second exhibition **must** carry the following thirteen fields. An asset missing any of these is not eligible for v4.3 build or v4.4 freeze.

| # | Field | Description |
|---|---|---|
| 1 | **title** | The asset's official or descriptive title. |
| 2 | **creator** | Author / artist / photographer / institution. |
| 3 | **date** | Date or date range of creation; "unknown" is acceptable if noted. |
| 4 | **institution** | The holding institution or publisher. |
| 5 | **collection URL** | Canonical page for the collection (not the image). |
| 6 | **image URL** | Direct image URL, or IIIF image service URL. |
| 7 | **rights statement** | The exact rights string from the source page (e.g., "CC0", "Public Domain", "CC BY 4.0"). |
| 8 | **identifier** | A persistent identifier (object ID, accession number, BHL page ID, etc.). |
| 9 | **local file path** | Repository path to the local copy, once approved. |
| 10 | **alt text** | Descriptive alt text for accessibility. |
| 11 | **caption** | Display caption used in the artifact card. |
| 12 | **source note** | Short free-text note on *how* the asset was located and verified. |
| 13 | **credit line** | The exact credit string that will appear in the rendered exhibition (e.g., "Biodiversity Heritage Library, Public Domain"). |
| 14 | **audit status** | One of `pending` / `verified` / `blocked` / `excluded` — set by v4.2 rights audit. |

The metadata table itself is *not* populated in v4.0. v4.0 only fixes the schema; v4.1 populates candidate rows; v4.2 promotes rows to `verified` or downgrades to `blocked` / `excluded`.

## Source acceptance rule

An asset may enter a formal exhibition (i.e., move past the v4.2 audit and into v4.3 build) **only if all** of the following are true:

1. **The source URL is reachable.** A `curl` or equivalent HTTP fetch returns a `200` (or `3xx` followed by `200`) at the moment of acceptance. The fetch result is logged in the asset's `source note`.
2. **The rights statement is locatable on the source page.** The exact string is copied into the asset's `rights statement` field. "Assumed public domain" is **not** an acceptable substitute.
3. **A credit line is writable.** The credit line composes cleanly from the institution name + rights statement + identifier, with no missing pieces.
4. **The source note is reconstructable.** Another person, given only the source note, can re-derive the asset's path from the source URL without guessing.
5. **No "search-result import" or "memory import".** The asset's provenance goes through a real, citable source page — not a Google Image result, not a Pinterest board, not a personal recollection of a previous browsing session.

If any of the five is not true, the asset is **not accepted** for the exhibition. It may remain in the candidate table with `audit status = blocked` or `excluded`, but it does not enter v4.3 build.

This rule is the single hard gate. The rights-risk register (companion doc) defines the *labels* that result from applying this rule; this rule defines the *conditions* under which the labels are applied.
